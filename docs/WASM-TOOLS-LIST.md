# Comprehensive WebAssembly Tools Reference

A curated list of tools and utilities that can be compiled to WebAssembly and run in the browser. These tools are designed to work in a sandboxed environment without direct network or filesystem access - they accept file/data inputs and produce outputs.

## Selection Criteria

Tools in this list must:
- Be compilable to WASM (WASI or Emscripten)
- Work without direct network access
- Work without direct filesystem access (use virtual FS)
- Accept input data and produce output data
- Be useful for development, data processing, or content creation

---

## Build Interface Classification

Every tool in this document is labeled with its **build interface** â€” how it was compiled to WebAssembly. This matters because WASI modules use a standardized system interface and can run in any WASI-compatible runtime, while Emscripten modules require JavaScript glue code specific to their build.

| Label | Build Interface | What It Means |
|-------|----------------|---------------|
| ðŸŸ¢ **WASI** | `wasm32-wasi` / `wasm32-wasip1` via wasi-sdk or Rust | Uses the standardized WASI system interface. No JS glue required. Runs in any WASI runtime (Wasmtime, Wasmer, browser WASI polyfill). **Co-do's native WASM runtime supports these directly.** |
| ðŸŸ¡ **Emscripten** | Emscripten (`emcc`) | Compiles C/C++ to WASM with Emscripten's JS glue layer. Requires Emscripten's virtual filesystem, memory management, and runtime. **Needs a bridge/adapter in Co-do.** |
| ðŸ”µ **wasm-bindgen** | `wasm32-unknown-unknown` via Rust `wasm-pack` / `wasm-bindgen` | Rust compiled to bare WASM with JS bindings generated by wasm-bindgen. Not WASI â€” uses custom JS imports. **Needs a JS interop layer.** |
| ðŸŸ  **Go JS** | `GOOS=js GOARCH=wasm` (Go) | Go compiled to WASM targeting the JavaScript host. Requires Go's `wasm_exec.js` runtime shim. **Not WASI. Needs Go's JS runtime.** |
| ðŸŸ£ **Go WASI** | `GOOS=wasip1 GOARCH=wasm` (Go 1.21+) | Go compiled to genuine WASI. Uses the WASI system interface. **Co-do's native WASM runtime supports these directly.** |
| âšª **Bare WASM** | `clang --target=wasm32` (no libc, no WASI) | Compiled directly with Clang to minimal WASM. No system interface â€” pure computation only. **Runs anywhere, but limited to pure functions.** |
| ðŸ”´ **GHC WASM** | GHC WebAssembly backend (Haskell) | Haskell compiled via GHC's WASM backend. Experimental, large output. **Needs custom runtime support.** |

### Why This Matters for Co-do

Co-do's WASM tool runtime natively supports **WASI** modules (ðŸŸ¢ and ðŸŸ£). These run directly in the sandboxed Web Worker with WASI syscall interception via the VFS layer. Tools using other build interfaces (ðŸŸ¡ Emscripten, ðŸ”µ wasm-bindgen, ðŸŸ  Go JS) require bridge code or adapters and may have larger bundle sizes or different initialization patterns.

**When choosing tools to integrate, prefer WASI-compliant modules** â€” they are simpler to sandbox, smaller in bundle size, and consistent in their interface.

## Implementation Notes

### WASM Compilation Approaches

1. **ðŸŸ¢ WASI (WebAssembly System Interface)** â€” **Preferred for Co-do**
   - Standardized syscall interface (file I/O, stdio, env vars, clock)
   - No JavaScript glue code needed
   - Smaller output, consistent interface
   - **Build with:** wasi-sdk (C/C++), `wasm32-wasip1` (Rust), `GOOS=wasip1` (Go 1.21+), SwiftWasm
   - **Co-do support:** Native â€” runs directly in WASM Worker with VFS syscall interception
   - **Limitations:** No threading (wasip1), no networking, limited process spawning

2. **ðŸŸ¡ Emscripten** â€” Most existing ports use this
   - Good for complex C/C++ projects (FFmpeg, SQLite, ImageMagick)
   - Provides virtual filesystem, pthread emulation, memory management
   - Larger output (includes JS glue + runtime)
   - **Co-do support:** Requires bridge/adapter code per-tool
   - **Limitations:** Each tool needs custom integration, larger bundles

3. **ðŸ”µ wasm-bindgen** (Rust `wasm32-unknown-unknown`)
   - Rust crates compiled with wasm-pack/wasm-bindgen
   - JS bindings auto-generated, good for library APIs
   - **Co-do support:** Needs JS interop layer
   - **Limitations:** Not a CLI interface â€” requires API-style integration

4. **ðŸŸ  Go JS** (`GOOS=js GOARCH=wasm`)
   - Go's original WASM target, requires `wasm_exec.js` runtime
   - **Not recommended** â€” prefer `GOOS=wasip1` for WASI compatibility
   - **Co-do support:** Needs Go's JS runtime shim

5. **ðŸŸ£ Go WASI** (`GOOS=wasip1 GOARCH=wasm`) â€” **Preferred for Go tools**
   - Genuine WASI, Go 1.21+ required
   - **Co-do support:** Native â€” same as ðŸŸ¢ WASI
   - **Limitations:** Goroutine blocking on some syscalls, no networking

6. **âšª Bare WASM** (clang `--target=wasm32`, no libc)
   - Minimal WASM for pure computation (hash functions, codecs)
   - Smallest possible output, fastest startup
   - **Co-do support:** Needs custom memory/function imports
   - **Limitations:** No filesystem, no stdio â€” pure functions only

3. **Rust to WASM**
   - Native wasm32-unknown-unknown or wasm32-wasi targets
   - Excellent tooling support (wasm-pack, wasm-bindgen)
   - Many CLI tools already have Rust ports

4. **AssemblyScript**
   - TypeScript-like syntax compiling to WASM
   - Easy onboarding for JS developers
   - npm install workflow

### Sandboxing Considerations

1. **Memory limits** - Set per-tool caps (e.g., 256MB default)
2. **Execution timeout** - Kill long-running tools (30s default)
3. **Virtual filesystem** - All file I/O through VFS layer
4. **No network** - WASI network imports not provided
5. **No threads** - Single-threaded execution (SharedArrayBuffer optional)

## Technical Challenges & Solutions

Porting software to WebAssembly involves overcoming several architectural constraints:

### 1. Single-Threaded Constraint

**Problem**: Browsers are event-driven. Legacy apps use blocking I/O or `sleep()` calls that freeze the UI.

**Solutions**:
- **Asyncify** (Emscripten): Instruments code to pause/resume execution, simulating blocking without freezing
- **Web Workers**: Move processing off main thread; use `SharedArrayBuffer` + `Atomics` for communication
- **Example**: vim.wasm uses Asyncify or SharedArrayBuffer to handle Vim's blocking main loop

### 2. File System Access

**Problem**: Native apps assume POSIX filesystem access. Browsers restrict this for security.

**Solutions**:
- **MEMFS**: Emscripten's in-memory filesystem (ephemeral)
- **IDBFS**: Maps to IndexedDB for persistence (slower)
- **OPFS**: Origin Private File System - high-performance persistent storage (modern browsers)
- **Example**: SQLite official build uses OPFS for near-native database performance

### 3. Networking Restrictions

**Problem**: Wasm modules cannot open arbitrary TCP/UDP sockets in browsers.

**Solutions**:
- **WebSocket tunneling**: Route traffic through WebSocket proxy
- **WebRTC**: P2P connections via Pion or similar
- **Fetch API bridge**: HTTP requests via JavaScript interop
- **Example**: postgres-wasm uses WebSocket proxy for external communication

### 4. Memory Constraints

**Problem**: 32-bit Wasm has ~4GB address limit; browsers often restrict further.

**Solutions**:
- **Streaming processing**: Process data in chunks rather than loading entirely
- **Memory growth**: Configure `ALLOW_MEMORY_GROWTH` in Emscripten
- **Example**: ffmpeg.wasm has ~2GB file size limit due to addressing constraints

### 5. Multi-Process Architecture

**Problem**: Apps like PostgreSQL fork processes; browsers don't support `fork()`.

**Solutions**:
- **Single-process mode**: PGlite uses Postgres's bootstrapping mode
- **Full virtualization**: postgres-wasm runs Linux in x86 emulator (v86)
- **Web Workers as processes**: Simulate process isolation with workers

### 6. Security Headers for Advanced Features

**Problem**: SharedArrayBuffer requires specific security headers after Spectre mitigations.

**Required headers**:
```
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
```
